## Scope
- In scope: the Individual Disbursement Process Summary page вҖ” summary query, CAD Maker submit, CAD Checker return, CAD Checker authorize, T24 file generation/upload, T24 deal result refresh, and three report downloads.
- The contract is brownfield: existing refactor endpoints are kept as RPC-style POST endpoints.
- Out of scope (Non-Goals): EPROISU0920/0921 data-entry authoring, async batch scheduler ownership, and final T24 field-by-field signoff beyond the deltas governed here.

## Assumptions / Dependencies / Constraints
- Caller identity (checker `empId`) is supplied by the authenticated platform context (session/JWT or equivalent), never by the client request body.
- The shared notification mechanism owns email delivery, send status, template rendering, and retry; this bundle owns only notification-task persistence.
- Valid disbursement currency is USD or KHR only (A-5 owner decision); any other currency is by-design unreachable.
- The T24 composition primary source is refactor-spec `funcisut24authorize` v1.5; legacy/backend parity is fallback baseline only where v1.5 is silent or ambiguous.

### R1 Summary initialization - еј·еҲ¶й»һ: both
covers-prd: FR-INIT-001, FR-INIT-002, FR-INIT-003, FR-UI-004, AC-001

Given a valid `applicationNo`, `epl-info-isu-summary` must return summary header, collateral, other collateral, role, current `CASE_PROGRESS`, CAD Checker option list `empList`, `t24ResultTime`, T24 status/result fields, and display data needed by the Summary page. When `CASE_PROGRESS=24` and the user has Maker role `404`, `empList` must contain active role-`405` CAD Checker employees only; inactive employees or non-405 roles must not be selectable for submit. BE must read from `TB_LON_SUMMARY_INFO`, `TB_DISBUR_DATE`, `TB_DISBUR_COLL`, `TB_DISBUR_OTHER_COLL`, `TB_T24_MAIN_BORROWER_INFO`, and `TB_T24_CO_BORROWER_INFO` as applicable. `TB_LON_SUMMARY_INFO.SECURE_ATTRIBUTE` is carried as server-read routing and audit context for Summary. EPROISU0922 does not create a separate S/U branch inside the Summary page: upstream ISU routing and collateral-page availability already decide whether secured collateral data is applicable. BE must not accept any client-supplied secure attribute as authority, and all Summary/authorize rules in this bundle apply to both `S` and `U` individual cases unless a later PRD/SRS delta explicitly introduces an S/U split.

Error-code contract: blank `applicationNo` returns `COMMON_MSG_ERROR_LON`; query/data failure returns `MSG_QUERY_FAIL`; no summary data returns `MSG_DATA_NOT_FOUND`. [evвҶ’R1]

### R3 Maker submit - еј·еҲ¶й»һ: BE
covers-prd: AC-002, AC-003

`epl-case-isu-summary-submit` must validate `applicationNo`, selected CAD Checker employee, and fresh T24 borrower/co-borrower check-date state before submit. On success, BE updates `TB_LON_SUMMARY_INFO.CASE_PROGRESS` to `25`, sets `CADC_CODE` and current handler for the checker, inserts `TB_APP_HISTORY`, and creates the notification task in `TB_NOTIFICATION_INFO` or the current notification mechanism. The notification recipients must include the selected checker and any effective proxy users from `TB_EMP_PROXY`; actual email delivery, status `S/F`, template rendering, and retry are owned by the shared notification mechanism and are not synchronous success criteria for this endpoint after the task is persisted. The state update, history insert, and notification-task creation must be atomic.

Submit must preserve the submit-Maker identity for the later four-eyes authorize guard. If `TB_LON_SUMMARY_INFO.CURRENT_USER_ID` is overwritten to the selected checker during submit, BE must still retain the pre-submit Maker identity through the existing state/history/audit data available to the authorize service; the authorize comparison must not accidentally compare the checker against the post-submit current handler.

Error-code contract: blank `applicationNo` returns `COMMON_MSG_ERROR_LON`; stale T24 borrower/co-borrower check date on submit returns the bare legacy key `EPROIS0922_CLICK_CHECK`; other validation failure returns a 400 platform validation response; submit failure returns `MSG_SUBMIT_FAIL`. [evвҶ’R3]

### R5 Checker authorize transaction - еј·еҲ¶й»һ: BE
covers-prd: AC-006, AC-007

`epl-case-isu-summary-auth` must run as one transactional business operation around validation, T24 file generation, SFTP upload, `TB_MESS_UPLOAD_SFTP_RECORD` insert, `TB_LON_SUMMARY_INFO` update, and `TB_APP_HISTORY` insert. On successful SFTP upload, BE sets `TB_LON_SUMMARY_INFO.CASE_PROGRESS` to `26`, clears current handler, and writes `DISBURSING_DATE`. It must not move to `27` until T24 deal result is confirmed. T24 file generation failure or blank `t24UpPath` returns safe category `T24_FILE_GENERATION_FAIL`; SFTP connection failure returns `SFTP_CONNECT_FAIL`; SFTP upload/file-transfer failure returns `SFTP_UPLOAD_FAIL`. These client-facing categories must not include host, user, credential, full path, stack trace, or raw customer data; the generated file name may be included for upload/file-transfer failure only. All three failure classes must stop before Summary `26`, `TB_MESS_UPLOAD_SFTP_RECORD`, and success history are committed, or must roll back to a consistent pre-authorize state.

Before any T24 file generation or SFTP action, authorize must enforce state `CASE_PROGRESS=25`, CAD Checker role `405`, assigned-checker/case ownership, and four-eyes separation. If the authenticated checker employee id equals the Maker who submitted the case into `CASE_PROGRESS=25`, BE must reject the request with named code `EPROIS0922_SELF_APPROVAL` and leave Summary, history, T24 file, SFTP upload record, and generated file side effects unchanged. This guard is service-level mandatory even if the user has endpoint seed access, both roles in test data, or a visible button.

Caller identity for authorize, including checker `empId`, must come from the authenticated context (session/JWT or equivalent platform principal). It must not be accepted from `ApplicationLangRequest` or any other client-supplied request body field, and the authorize request DTO must remain limited to application/language context.

Error-code contract: blank `applicationNo` returns `COMMON_MSG_ERROR_LON`; stale T24 borrower/co-borrower check date on authorize returns the bare legacy key `EPROIS0922_AUTHORIZE_CHECK`; self-approval/four-eyes violation returns `EPROIS0922_SELF_APPROVAL`; exchange-rate lookup failure is carried as `FAILED_E304` with bare key `EPROIS0921_UI_RAET_FIND_ERROR`; authorize failure returns `MSG_AUTHORIZE_FAIL`; T24 file generation or blank path failure returns `T24_FILE_GENERATION_FAIL`; SFTP connection failure returns `SFTP_CONNECT_FAIL`; SFTP upload/file-transfer failure returns `SFTP_UPLOAD_FAIL`. [evвҶ’R5]

### R6 T24 prerequisite checks - еј·еҲ¶й»һ: BE
covers-prd: AC-004, FR-INIT-005

Before submit or authorize, BE must verify the latest T24 borrower and co-borrower `CHECK_DATE` values from `TB_T24_MAIN_BORROWER_INFO` and `TB_T24_CO_BORROWER_INFO`. Every required `CHECK_DATE` must equal the system calendar date; there is no cross-day grace period and no business-day/holiday exception. If the main borrower or any co-borrower check date is not the system calendar date, the operation must fail before state mutation. This carries the refactor `funcConfCheckDate` behavior and prevents stale borrower information from driving a T24 file.

Error-code contract: stale check date returns `EPROIS0922_CLICK_CHECK` for submit and `EPROIS0922_AUTHORIZE_CHECK` for authorize; query/data failure returns `MSG_QUERY_FAIL`. [evвҶ’R6]

### R7 T24 import file composition - еј·еҲ¶й»һ: BE
covers-prd: AC-006

`funcIsuT24Authorize` must compose T24 records using refactor-spec `funcisut24authorize` v1.5 as the primary T24 composition source. Parity evidence is fallback baseline only where v1.5 is silent, ambiguous, or explicitly escalated for owner confirmation. The public response exposes only `t24UpPath`; individual T24 A-H record fields are internal file content and QA evidence, not client-editable API fields. T24 file format: field delimiter is `!@#`, multi-value delimiter is `|`, record separator is CRLF, and outbound file encoding follows legacy behavior by not specifying an explicit charset (runtime JVM default charset). Do not hard-code UTF-8 unless a later owner decision supersedes legacy parity. Records must preserve max lengths and reject missing required source data before upload. Missing required T24 source data, including required branch-profile identity fields such as `T24_COMPANY`, must return `FAILED_E999` before SFTP upload or Summary success mutation.

The file must source monetary and rate fields from `TB_DISBUR_DATE`, `TB_LOAN_CONDITION_DETAIL`, `TB_LOAN_CONDITION_FEE`, and related T24 borrower/account tables. Null/default policy follows v1.5 explicit field rules: do not generalize all nulls to `N/A` or all nulls to empty string. `NAME_OF_BOREY` and `CUBC_RELATED_CIF_NO` null/blank values output `N/A`, while `HOUSING_PROJECT_CODE` null/blank outputs empty string. The closed itemized T24 A-H field deltas are:

- A15: null/blank `HOUSING_PROJECT_CODE` outputs empty string (not `N/A`).
- A16 loan type literal: `PRODUCT_CODE=01` and `MTG_PRODUCT_CODE=02` outputs `NORMAL.LAON`; this is intentional and must not be corrected to `NORMAL.LOAN`.
- A31 and G7 `AGREEMENT_NO`: output the rightmost 16 characters when the source exceeds 16 characters; values вүӨ16 characters are preserved. This truncation applies only to these two fields.
- A52 `DISTRICT_NAME`: read borrower district name from `TB_DISTRICT.DISTRICT_NAME` using the main borrower's `RESIDENT_PROVINCE_CODE` plus `RESIDENT_DISTRICT_CODE` and the existing disbursing-date `UPD_DATE` selection convention; no district match outputs empty string.
- Segment B (non-B8): `COLLATERAL.RIGHT`, borrower T24 no, blank limit product, collateral code, blank collateral type, application no, and transaction branch (legacy/backend parity).
- B8 and C9: source from `TB_BRANCH_PROFILE.T24_COMPANY` for the disbursing department; follow legacy/current source with no separate T24 contract override.
- C12: read sequence-specific `TB_COLL_VALUE_INFO.SUG_VAL` by `APPLICATION_NO` plus current `COLL_DATA_SEQ`; must not read `TB_COLL_INFO.SUG_VAL` or the generic C-row collateral map.
- C13: read `TB_LON_SUMMARY_INFO.DECISION_DATE` formatted as `YYYYMMDD`; must not use `TB_T24_MAIN_BORROWER_INFO.CHECK_DATE`.
- C20: read the current C row's `TB_DISBUR_COLL.INS_EXPIRY_DATE`, format with the T24 date rule, output empty string when absent; must not use non-schema `INS_END_DATE`.
- C26 `TITLE_DEED_NO`: read only `TB_COLL_TITLE_DETAIL.TITLE_DEED_NO` rows for the current C row's `APPLICATION_NO` plus `COLL_DATA_SEQ`, join multiple rows with `|`; must not pre-merge all application title deeds into every C row.
- Segment D: `CUSTOMER.SSI`, borrower T24 no, disbursement currency, drawdown account, transaction branch, `ALL`, and application no.
- E21 `CHRG_AMOUNT`: read `TB_LOAN_CONDITION_FEE.CBC_FEE`, `TB_DISBUR_DATE.DISBURSEMENT_CURRENCY`, `EX_RATE_BUY`, and `EX_RATE_SELL`; output raw `CBC_FEE` for `USD`; output `CBC_FEE * ((EX_RATE_BUY + EX_RATE_SELL) / 2)` for `KHR` using backend average scale 4 `HALF_UP` and final scale 0 `HALF_UP`; output `0` for any other currency only as a by-design-unreachable fallback under A-5. Do not restore legacy all-non-USD exchange conversion unless A-5 is reopened.
- G4 and H4: output `TB_DISBUR_DATE.DISBURSEMENT_CURRENCY`. G10 and H8: decide the rate field by `TB_DISBUR_DATE.DISBURSEMENT_CURRENCY`, not by `TB_MAIN_BORROWER_ACC.CURRENCY`. `USD` outputs empty string; `KHR` outputs `TB_DISBUR_DATE.EX_RATE_BUY`; any non-USD-non-KHR value remains by-design unreachable under A-5.
- G11/G12 law-firm fee remark mapping: note `00` uses `Registration Fee`, `01` uses `De-register fee`, `02` uses `Due diligent fee`, `03` uses `Ownership change`, and `90` uses `LAW_FIRM_90_OTHER_REMARK`; do not use `Legal fee`, `Other fee`, or `Special fee` for these notes.
- Segment H1-H7: EIR/refinancing fee rows using transaction types `FT.LOAN.EIR.52153`/`FT.LOAN.EIR.52155`, branch, drawdown account, disbursement currency, fee amount, credit account, and application no.

Newly discovered field deltas must be opened as new named pending items rather than reviving a generic umbrella. [evвҶ’R7]

### R9 T24 deal result refresh - еј·еҲ¶й»һ: BE

### R10 Monetary precision, truncation, and exchange-rate handling - еј·еҲ¶й»һ: BE
covers-prd: T24 monetary fields

Agreement numbers and T24-specific remarks must be truncated or rejected only according to T24 field-specific evidence, not arbitrary UI truncation.

Precision and truncation matrix:

| `TB_DISBUR_DATE.EX_RATE_BUY`, `EX_RATE_SELL` | BE reads persisted DB rate values as authoritative for Summary/T24; any derived average uses explicit rounding mode and scale, currently `HALF_UP` scale 4 for the KHR average path. No hidden conversion from client input. |

### R11 Auditability and transaction consistency - еј·еҲ¶й»һ: BE
covers-prd: NFR audit/data consistency

All mutating operations must write audit/history records after successful state transitions and must avoid partial commits. Submit and return must roll back summary state and history if notification-task creation fails; async email delivery failure after the task is persisted is handled by the shared notification mechanism and does not roll back the already-committed submit/return state. Submit, return, authorize, and T24 result update must have deterministic rollback behavior across summary state, history, notification/SFTP logs, and generated/uploaded files.

Logging must not write complete request or response bodies. T24 import payload content, `DRAWDOWN_ACCOINT` / `drawdownAccount`, and `DISBURSEMENT_AMOUNT` / `disbursementAmount` are customer data and must be masked or omitted in application, integration, and error logs. Logs may record correlation ids, endpoint ids, safe result codes, and generated file names, but must not log credentials, full secret paths, stack traces, host/user secrets, or raw customer data beyond existing safe audit policy. [evвҶ’R11]

### R12 Report downloads - еј·еҲ¶й»һ: both
covers-prd: AC-010
