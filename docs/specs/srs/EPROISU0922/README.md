## 這頁在幹嘛（一句）
個金案件撥貸前的**最後一關**：CAD 經辦（Maker）把撥貸彙總送審、CAD 主管（Checker）覆核後**授權撥貸**——授權瞬間系統會**換匯算金額、組 T24 匯入檔、上傳 SFTP**，再等 T24 回拋交易結果決定成敗。全程走**四眼（Maker-Checker）**控制。

## 怎麼運作〔此段隨頁型態而異、非通用範本〕
撥貸彙總是「**狀態機 + 端到端整合**」頁，不是計算頁也不是單純存草稿頁：

1. **載入**（`R1`/`R2`）：依案件目前進度與使用者角色，回傳彙總抬頭、擔保品、可選的覆核主管清單、T24 狀態/結果，並算出「T24 結果是否可刷新」旗標。畫面按鈕/區塊**全由後端授權 + 案件狀態**決定，前端藏按鈕不算數。
2. **送審 / 退回**（`R3`/`R4`）：
   - **送審**＝經辦選定覆核主管 → 案件進到「待覆核」、建通知任務（含代理人）。
   - **退回**＝主管退回給經辦、回到「可編輯」狀態，**不會碰 T24**。
3. **授權**（`R5`，本頁核心）：主管按授權後，後端在**單一交易**內依序做：四眼/角色/狀態檢核 → 換匯 → 組 T24 匯入檔 → 上傳 SFTP → 寫上傳紀錄 → 更新案件到「已撥/待 T24 結果」、寫撥款日。任一步失敗都要**整筆回滾**，且錯誤訊息不得洩漏主機/帳密/路徑/堆疊。
4. **T24 結果刷新**（`R9`）：線上讀回 T24 回拋結果。**成功**→ 案件進到「完成」；**任何非成功**→ 進到「失敗」狀態並寫結案原因。對已記錄的結果具**幂等性**（不重複寫入）。
5. **報表下載**（`R12`）：彙總 / 交易結果 / 訊息碼紀錄三張 PDF，純讀、不改狀態。

## 規則速覽（白話；精確值見對應 `Rn`）
**主流程（依執行序）**
1. **載入** `R1`/`R2`：回彙總顯示資料 + 可選覆核主管（只列在職、對的角色）+ T24 結果可刷新旗標；查詢**不得改狀態**；畫面權限以後端為準。
2. **送審** `R3`：驗經辦/選定主管/檢查日 → 轉「待覆核」、建通知任務（含代理人）。狀態+歷程+通知任務**同生同滅**。
3. **退回** `R4`：主管退回 → 回「可編輯」、建通知任務；**絕不產生或上傳 T24 檔**。
4. **檢查日前置** `R6`：送審/授權前，借款人與所有共借人的檢查日**都必須等於系統當日**（無跨日寬限、無假日例外），否則在改狀態前就擋下。
5. **授權交易** `R5`/`R7`：**單一交易**內換匯→組 T24 檔→上傳 SFTP→寫上傳紀錄→轉「已撥」、寫撥款日；尚未拿到 T24 結果前**不得**進「完成」。
6. **T24 結果** `R9`：成功→「完成」；任何非成功→「失敗」狀態 + 寫結案原因；對已記錄結果**幂等**。
7. **報表** `R12`：三張 PDF 純讀、不改狀態。

**橫切（每階段都適用）**
- **四眼控制** `R5`/`R8`：**送審的經辦不可當覆核主管授權同一案**——即使他同時有兩種角色、seed 權限或看得到按鈕，後端 service 層**必擋**（這是本頁最關鍵的安全控制）。
- **授權/雙層** `R8`：mutating endpoint 一律**平台 API 授權 + 後端 service guard**（驗案件/角色/狀態/直呼）；**只設 API seed 不夠**。被拒時所有寫入（狀態/歷程/通知/SFTP/T24 檔）保持不動。
- **身分來源** `R5`：覆核者身分**只能**取自登入情境（session/JWT），**不可**從 request body 帶入。
- **錯誤/回滾** `R5`/`R11`：T24 組檔失敗、SFTP 連線失敗、SFTP 上傳失敗各有**安全錯誤分類**（不含主機/帳密/路徑/堆疊/客戶資料）；任一失敗在「已撥」狀態與上傳紀錄落地前停住或整筆回滾。
- **換匯** `R7`/`R10`：撥貸幣別**只支援 USD / KHR**（A-5 決策）；KHR 走買賣匯率平均（精確 rounding/scale 見 `R10`），其他幣別屬「設計上不可達」的 fallback，**不得**復活 legacy 全幣別換匯。

## NFR（精確見 spec `## NFR`）
安全（mutating 必雙層授權、四眼強制、seed 單獨不足）／交易一致（狀態轉移原子、不得部分提交 Summary/T24/SFTP/歷程）／精度（金額/匯率照 DB 精度，禁靜默溢位/截斷/只靠前端驗）／隱私（log 與錯誤訊息不得露 SFTP 帳密/完整路徑/堆疊/完整請求回應/T24 payload/撥款帳號/撥貸金額/真實客戶資料）。

## 資料表（確切表名/欄位/PK 見 `schema.sql`）
主狀態＝撥貸彙總狀態表（案件進度/現處理人/覆核人/撥款日）；T24 來源＝撥貸資料、放款條件/費用、擔保品/其他擔保品、估價、地區、分行資料、借款人/共借人 T24 與檢查日諸表；checkpoint/log＝SFTP 上傳紀錄、T24 結果紀錄、結案原因、歷程、通知、代理人。**T24 組檔的欄位來源踩雷多，逐欄正確來源見 spec `R7`/`R10` 與附錄。**

## ⚠️ 雷區（依 RD 接手順序）
**動工前必知**
- **四眼**：送審經辦≠授權主管，後端 service 層強制擋（client/seed/按鈕都不可信）；違反回專屬自核錯誤碼（見 `R5`）。
- **授權＝一筆端到端交易**：換匯+組檔+SFTP+狀態+紀錄同生同滅；中途失敗必回滾到授權前一致狀態，**別把案件留在半撥狀態**。
- **檢查日**：借款人/共借人檢查日**必須等於系統當日**，無寬限——送審與授權用**不同**的過期錯誤碼（見 `R6`）。
- **身分來源**：覆核者 `empId` 只能來自登入情境，**禁**從 request body 取。

**實作踩坑**
- **T24 欄位來源**：A-H 多欄有「正確來源 vs legacy 誤用來源」的明確差異（如某估價欄要讀序列專屬來源、不可讀通用合併表；某日期欄要讀核准日、不可讀檢查日；某標題欄只取當前擔保品序列、不可全案合併）——**逐欄照 `R7`/`R10`**，別憑直覺。
- **換匯/rounding**：KHR 走買賣均價，中間與最終 scale/rounding 模式有明定（見 `R10`），別自行四捨五入。
- **截斷規則**：某些 T24 欄取**最右 16 碼**、但**只限指定那兩欄**，不可套到其他欄；其餘欄超長要**擋下**、不得靜默截斷。
- **檔案格式/編碼**：分隔符與換行有明定，輸出編碼**沿用 legacy 執行期預設字集、別硬寫 UTF-8**（見 `R7`）。
- **幣別不可達分支**：非 USD/KHR 的 `0`/空白只是 A-5 設計性 fallback，別當成正常路徑、也別回填 legacy 行為。
- **必填來源缺漏**：T24 必要來源（含分行身分欄）缺漏要在上傳前回專屬失敗碼、不得放行。
- **通知非同步**：通知任務建立失敗才回滾送審/退回；任務落地後的 email 寄送由共用機制負責、不是本 endpoint 的同步成功條件。

**維護註記**
- 本包所有 TBD / REF-D **已全部 closed**（T24 umbrella + B/D/H 段 + 分行身分欄位來源已收斂），無 open pending 擋住任何 `Rn`；若日後發現新 T24 欄位 delta，須**開新具名 pending**，不得復活舊 umbrella。
- 撥款帳號實體欄名沿用 legacy 拼字錯誤（物理欄名保留、型別已 DB-resolvable 修正），對位時注意（見 spec DB delta）。
