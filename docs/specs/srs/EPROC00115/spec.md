## Rules
> 每條只寫 **to-be 契約**（系統最終應如何）；佐證/出處/as-is/決策 → 下半 `Rule Evidence`，用 `[ev→Rn]` 指過去。

### R1 Load option lists - 強制點: BE - covers-prd: REQ-001
The select endpoint must require `langType` and return an EPRO envelope whose `data` contains `facilityList`, `loanLimitTypeList`, `ccyList`, and `titleDeedTypeList` from `TB_COMMON_FIELD_OPTIONS` option codes `FACILITY_TYPE`, `LOAN_LIMIT_TYPE`, `CCY`, and `TITLE_DEED_TYPE`. Blank `langType` is a validation error carrying `E102`. [ev→R1]

### R2 Query exposure rows by application number - 強制點: BE - covers-prd: REQ-001, REQ-002
The info endpoint must require `applicationNo` and `isQuery`. It must validate the loan summary using the established query-mode semantics, then return an EPRO envelope whose `data.facilitiesList` has all `TB_GROUP_EXPOSURE` rows for the request `applicationNo` ordered by `DATA_SEQ`. If no exposure rows exist, return `data.facility=""` and `data.facilitiesList=[]`; the UI may create an empty visual row, but the API does not synthesize a row. [ev→R2]

### R3 Preserve facility value as header-level field - 強制點: BE - covers-prd: REQ-003
`facility` is required on save, max length 2, and represents one page-level `FACILITY_VALUE`. On save the backend writes the same request `facility` value to every inserted `TB_GROUP_EXPOSURE` row. The API response row object does not carry per-row `facilityValue`. [ev→R3]

### R4 Manage row add/delete and minimum one-row UI behavior - 強制點: FE - covers-prd: REQ-004
The UI must support adding and deleting exposure rows while preventing deletion of the final row. It must not call save with an unintended empty list caused by deleting all visible rows. `COMMON_MSG_ONE_DATA` is a UI-only prompt for this guard, not an API error response. The maximum row count must follow the legacy injected `top.groupSize` configuration for `com.cathaybk.epro.i0.module.EPROI0_0115.size`; neither FE nor BE may hard-code a different limit without a later change request. [ev→R4]

### R5 Calculate and display USD/KHR totals without a separate API - 強制點: FE - covers-prd: REQ-005
There is no current/refactor standalone calculate endpoint for EPROC00115. The UI must calculate visible totals client-side from the same form rows that are sent to save. The to-be display contract is fixed to USD and KHR totals for LCC and Outstanding only. `CCY` values outside USD/KHR must not be treated as supported total currencies for this page. LCC totals must bucket by `lccAmountCur`; Outstanding totals must bucket by `outstandAmountCur`. [ev→R5]

### R6 Save draft vs Finished validation and checkpoint polarity - 強制點: BE - covers-prd: REQ-006
Save and Finished use the same save endpoint. `isFinish=false` is draft Save and updates `EPROC00115` checkpoint to `Y`; `isFinish=true` is Finished and updates checkpoint to `N`. Save draft still validates request shape and numeric/date formats, but Finished adds row-level required-field checks. The conditional Outstanding/Maturity branch follows current/refactor behavior: Outstanding Amount and Maturity Date are required when `loanLimitType != "2"`. This retains the current contract with an explicit assumption that code `"2"` is the only non-outstanding loan-limit type. [ev→R6]

### R7 Save by full replacement in one transaction - 強制點: BE - covers-prd: REQ-007
Saving must delete existing `TB_GROUP_EXPOSURE` rows for the request `applicationNo`, insert the submitted list with `DATA_SEQ` assigned from 1 in request order, and update the checkpoint in the same transaction. Any failure after delete, including insert or checkpoint update failure, must roll back the delete and all inserts.

The backend must set `APPLICATION_NO` from request `applicationNo`, not from row content. [ev→R7]

### R8 Route checkpoint by CS/CU in the new schema - 強制點: BE - covers-prd: REQ-008
For the current new schema, corporate secured cases write `TB_CHECK_POINTS_CS.EPROC00115` and corporate unsecured cases write `TB_CHECK_POINTS_CU.EPROC00115`, using `TBLonSummaryInfo.LON_ATTRIBUTE + SECURE_ATTRIBUTE` as the branch key. Unsupported or missing summary attributes must fail with `E126` and must not write exposure rows. No separate RC checkpoint table is part of this SRS. [ev→R8]

### R9 Enforce auth, editability, safe errors, and cross-module consumers - 強制點: both - covers-prd: REQ-010
All three endpoints require platform API authorization using their exact `TB_API_AUTH.API_ID` rows. Mutating calls must also enforce backend-owned case/edit authorization before DB writes; UI hidden/disabled controls are not sufficient. Error handling must carry PRD/current platform codes where applicable: `E102`, `E126`, `E130`, `E201`, `E405`/access denied, `E498`, `E999`, `MSG_DATA_NOT_FOUND`, `MSG_OVER_COUNT_LIMIT`, `MSG_QUERY_FAIL`, `COMMON_MSG_ERROR_LON`, and `COMMON_MSG_SAVE_FAIL`. Current successful API responses use the EPRO envelope `code=0000`, `message=Success`, and endpoint-specific `data`; save currently returns an empty `data` object. Current `ReturnEnum` business errors use the normal EPRO response envelope; HTTP 401 is reserved for authorization or auth-context failures: `E405` means authorization denied, and `E498` means the auth/session context is invalid or unusable for the request. `COMMON_MSG_SAVE_SUCCESS` and `COMMON_MSG_TOTAL_FAIL` are legacy/UI message references and must not be modeled as raw API response bodies for these three endpoints. Safe logs must not include full sensitive payloads.

`TB_GROUP_EXPOSURE` is a shared data source for report and score modules; SRS changes must not alter columns, `DATA_SEQ` ordering, or LCC/Outstanding/Collateral field meaning without explicit cross-module decision. [ev→R9]

### R10 Final loan-limit-type semantics - 強制點: both - covers-prd: REQ-006
Outstanding Amount, Outstanding Currency, Tenor, Maturity Date, Collateral Currency, and Collateral Amount are required on Finished when `loanLimitType != "2"`. This deliberately follows the established backend/frontend/refactor behavior. The assumption is that code `"2"` is the only loan-limit type that does not require the Outstanding/Maturity branch; changing that domain meaning requires a later owner change. [ev→R10]

### R11 Final third-currency totaling rule - 強制點: FE - covers-prd: REQ-005
The to-be contract supports only USD and KHR visible totals. Rows with `CCY` values outside USD/KHR must not be counted into a hidden third bucket for this page. If such a value is accepted by the option source, the page must either reject it before save/query totals are committed or display it as unsupported; it must not silently fold into KHR. [ev→R11]

### R12 Final row-limit source - 強制點: FE - covers-prd: REQ-004
The row limit source is the legacy injected `top.groupSize` value, configured from `com.cathaybk.epro.i0.module.EPROI0_0115.size` in the parent JSP. The SRS intentionally does not invent a literal number. New UI and backend validation must bind to the same platform/config value or an explicitly migrated equivalent. [ev→R12]

### R13 Final blank loan-limit remediation - 強制點: both - covers-prd: REQ-009
If persisted exposure rows contain blank `LOAN_LIMIT_TYPE`, the info endpoint must reject the query with a controlled business error that forces data remediation before the page can be treated as complete. The new contract does not carry a silent `queryGroupExposure` flag and does not mark the parent tab complete while blank legacy data remains. [ev→R13]

### R14 Final RC 0215 checkpoint mapping - 強制點: BE - covers-prd: REQ-008
RC 0215 is intentionally unified into the new CS/CU checkpoint columns. Finished and draft saves for the modern EPROC00115 page write `TB_CHECK_POINTS_CS.EPROC00115` or `TB_CHECK_POINTS_CU.EPROC00115` according to the same branch logic as R8. No `EPRO_TB_CHECK_POINT_RC_CORP/CU` target is recreated by this SRS. [ev→R14]

### R15 Final RC old-case Finished behavior - 強制點: both - covers-prd: REQ-006, REQ-008
The modern EPROC00115 page does not preserve a separate RC old-case mode. Finished remains available under the normal editability and validation rules, and checkpoint update follows R8/R14. Any upstream old-case restriction must be enforced by the frame/workflow layer before this page is entered. [ev→R15]

### R16 Final report collateral currency rule - 強制點: BE - covers-prd: REQ-010
To-be report/read consumers must use `COLLATERAL_CUR` for collateral currency and must not copy the wrong outstanding-currency source. This is the explicit cross-module decision required by R9.

R9 cross-module impact: `TB_GROUP_EXPOSURE` column meanings remain unchanged for all score/report consumers. Report-module owners for CS0181/CU0181 must align their collateral-currency reads to `COLLATERAL_CUR` or record a report-module SRS exception; this EPROC00115 bundle creates no exception and does not change DB column semantics. [ev→R16]

### R17 Final facility label - 強制點: FE - covers-prd: REQ-003
The UI label for `facility` / `FACILITY_VALUE` is `Group/Exposure` for this page. The API and DB names stay unchanged, and the select source remains `FACILITY_TYPE`. [ev→R17]

### R18 Final route/menu naming - 強制點: both - covers-prd: REQ-001
The modern page identity is `EPROC00115`. Current feature inventory and page mapping use `EPROC00115`, and `TB_PAGE_MENU` is the route/menu data source. No separate c0 borrower-group-exposure page code is introduced by this SRS. [ev→R18]

### R19 Final facilities-list backend validation - 強制點: BE - covers-prd: REQ-006, REQ-007
The to-be OpenAPI contract requires `facilitiesList` on save. Backend validation must reject a missing or null list before service iteration and return a controlled EPRO validation response. The SRS contract remains strict rather than weakening the request schema. [ev→R19]

### R20 Final save service authorization - 強制點: BE/security - covers-prd: REQ-010
Save/Finished authorization is two-layer: platform `TB_API_AUTH` must allow the exact save API, and the service layer must independently enforce case ownership/editability before delete/insert/checkpoint mutation. FE-disabled controls and API seed rows alone are insufficient. [ev→R20]

### R21 Final FE numeric integer limit alignment - 強制點: FE - covers-prd: REQ-005, REQ-006, REQ-010
The to-be contract follows backend/DB/refactor precision: `LCC_AMOUNT`, `OUTSTAND_AMOUNT`, `COLLATERAL_AMOUNT`, `INTEREST_RATE`, and `LTV` allow 15 integer digits plus 2 decimals. FE numeric input must align to the 15+2 contract, unless a later approved change narrows both API and DB-facing validation. [ev→R21]

## NFR
- Transaction: save/Finished delete+insert+checkpoint update must commit or roll back together; any failure after delete rolls back the delete and all inserts (R7).
- Security: all three endpoints require platform `TB_API_AUTH`; mutating save additionally requires a service-level case/edit guard before DB writes; seed/API authorization alone is insufficient (R9, R20).
- Observability: safe logs must not include full sensitive payloads.
- Cross-module stability: `TB_GROUP_EXPOSURE` columns, `DATA_SEQ` ordering, and LCC/Outstanding/Collateral field meaning must not change without an explicit cross-module decision (R9, R16).
