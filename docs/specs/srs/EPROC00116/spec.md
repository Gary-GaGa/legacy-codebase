## Rules
> 每條只寫 **to-be 契約**（系統最終應如何）；佐證/出處/as-is/決策 → 下半 `Rule Evidence`，用 `[ev→Rn]` 指過去。

### R1 Load options and current-case data - 強制點: both
covers-prd: FR-001

The page must call select and info endpoints when loading EPROC00116. Select requires `applicationNo` and `langType` and returns option lists for `CCY`, `CCY_UNIT`, and `TYPE_OF_YEAR`. Info requires `applicationNo` and `isQuery`; it must query loan summary, `TB_FIN_STATEMENT_MAIN`, `TB_FIN_STATEMENT_BALANCE_GI`, `TB_FIN_STATEMENT_INCOME_GI`, and `TB_FIN_STATEMENT_CASHFLOW_GI`, then return the current-case maps plus `haveData`.

`haveData=Y` MUST require the main record plus all three GI detail groups needed for print/export: Balance, Income, and Cashflow. If any of the three detail groups is missing, including Cashflow-only missing, info MUST return `haveData=N` and FE MUST keep Print/Excel/PDF actions hidden. If the current case is not query mode and the loan summary/application date is missing, the API returns business error `E126` / `FAILED_E126`. [ev→R1]

### R2 Query reference application without mutating current case - 強制點: both
covers-prd: FR-002

The reference query endpoint must accept `reApplicationNo`, validate the reference case through the current repository rule, and return main, Balance GI, Income GI, and Cashflow GI maps only when all source data is present. The migrated eligibility rule is the current repository/application-date rule, which requires an eligible `TB_LON_SUMMARY_INFO` row with `CASE_PROGRESS in ('08','09','14','20','21','22','23','24','25','26','27','C1')` and non-null `APP_CON_TYPE`, then requires main plus all three GI detail groups. Missing loan-date/status or missing any financial statement group returns `FAILED_E116` and must not write to current-case tables or checkpoint. Error-key compatibility is resolved under R10. [ev→R2]

### R3 Maintain main record and five financial-highlight text areas - 強制點: both
covers-prd: FR-003

Main data must carry `applicationNo`, `currency`, `currencyUnit`, company/audit fields, three criteria flags, audited-FS answers, and five highlight text areas: `loanRepaymentHT`, `financialSituationHT`, `businessRiskHT`, `borrowerRiskHT`, and `summaryHT`. Current API fields allow each highlight section up to 6000 characters; persistence splits each section into `_1` and `_2` columns of 3000 characters each.

On Finished, `currency`, `currencyUnit`, company name, audited-FS answers, and the five highlight text areas are required. The migrated public contract is the five structured API fields only; legacy pipe-delimited `HIGHLIGHT` is not a public API field. Persistence source of truth is the five split column pairs (`LOAN_REPAYMENT_1/2`, `FINANCIAL_SITUATION_1/2`, `BUSINESS_RISK_1/2`, `BORROWER_RISK_1/2`, `SUMMARY_1/2`). `HIGHLIGHT_CLOB` is reserved for storing the raw legacy `HIGHLIGHT` string as migration provenance when such a raw value exists, and it is not the primary read source for current save/query.

Legacy migration from one raw `HIGHLIGHT` value MUST split by the first four literal pipe (`|`) delimiters into the five API fields in order: loan repayment, financial situation, business risk, borrower risk, and summary. Missing segments map to empty values. Records with fewer than four delimiters, more than four delimiters, or any segment exceeding 6000 characters MUST be flagged in the migration/test report for RD/DBA follow-up; because legacy provided no delimiter escaping, ambiguous embedded pipes cannot be losslessly inferred without preserving the raw value in `HIGHLIGHT_CLOB`. Implementation and migration tests are code-stage DoD, pending RD.

Currency formatting carries PRD BR-016: when `mainMap.currency = KHR`, FE amount inputs/display for Balance, Income, and Cashflow must use zero fraction digits; USD keeps decimal formatting. This is presentation/input formatting only and does not change DB numeric precision. [ev→R3]

### R4 Calculate and persist Balance Sheet GI rows - 強制點: BE
covers-prd: FR-004, FR-007

Balance rows are ordered by `DATA_SEQ`, use `BALANCE_DATE`, `PERIODS`, and `TYPE_OF_YEAR`, and persist the active GI columns in `TB_FIN_STATEMENT_BALANCE_GI`. Calculation must derive at least:

- `TOTAL_CUR_ASSETS` from current-asset fields.
- `NET_FIXED_ASSETS = LAND + BUILDING + EQUIPMENT + OTH_FIXED_ASSETS - DEPRECIATION_AMORTIZATION`.
- `NON_CUR_ASSETS` from fixed/non-current asset groups.
- `CUR_LIABILITIES`, `NON_CUR_LIABILITIES`, and `TOTAL_EQUITY` from their component fields.
- `TOTAL_ASSETS = TOTAL_CUR_ASSETS + NON_CUR_ASSETS`.
- `TOTAL_LIAB_EQUITY = CUR_LIABILITIES + NON_CUR_LIABILITIES + TOTAL_EQUITY`.
- `DIFFERENCE = TOTAL_ASSETS - TOTAL_LIAB_EQUITY`.

Physical DB/source typo fields MUST remain mapped as physical names in persistence: `AP_RELATED_PARITIES` and `SHARDHOLDERS_LOAN`. Public API/FE DTO names MUST use canonical business names such as `apRelatedParties` and `shareholdersLoan`; BE mapping is responsible for translating between the canonical JSON names and physical DB typo columns. Derived fields are backend-calculated and MUST NOT be trusted from the client payload on save.

Finished state must reject rows where required Balance fields are blank. Calculation/result messages must include the unbalanced-sheet condition when first-year totals are zero or `DIFFERENCE != 0`. [ev→R4]

### R5 Calculate and persist Income Statement GI rows - 強制點: BE
covers-prd: FR-005, FR-007

Income rows are ordered by `DATA_SEQ`, use the physical date column `IMCOME_DATE`, and persist the active GI columns in `TB_FIN_STATEMENT_INCOME_GI`. Calculation must derive:

- `GROSS_PROFIT = REVENUES - COST_OF_GOOD_SOLD`.
- `OPERATING_EXPENSES` from salary, commission, marketing, transportation, and utilities.
- `OPERATING_PROFIT = GROSS_PROFIT - OPERATING_EXPENSES`.
- `NON_OPERATING_INCOME` and `NON_OPERATING_EXPENSES` from their component fields.
- `PROFIT_BEFORE_TAX = OPERATING_PROFIT + NON_OPERATING_INCOME - NON_OPERATING_EXPENSES`.
- `ATTRIBUTED_TO_PARENT_COMP = CUR_PERIOD_PROFIT - MINORITY_INTERESTS`.

Physical DB/source typo fields MUST remain mapped as physical names in persistence: `IMCOME_DATE` and `IMPAIMENT_LOSS`. Public API/FE DTO names MUST use canonical business names such as `year`/`incomeDate` semantics and `impairmentLoss`; BE mapping is responsible for translating between the canonical JSON names and physical DB typo columns. Derived fields are backend-calculated and MUST NOT be trusted from the client payload on save. [ev→R5]

### R6 Calculate and persist Cashflow Statement GI rows - 強制點: BE
covers-prd: FR-006, FR-007

Cashflow rows are ordered by `DATA_SEQ`, use `CASHFLOW_DATE`, `PERIOD`, and `TYPE_OF_YEAR`, and persist the active GI columns in `TB_FIN_STATEMENT_CASHFLOW_GI`. Calculation must derive:

- For year N > 1, `OPENING_BALANCE(N)` from previous year `END_BALANCE`.
- `END_BALANCE = NET_I_D_IN_CASH + OPENING_BALANCE`.
- `OPERATING_NET_CASHFLOW = NET_I_D_IN_CASH - FINANCING_NET_CASHFLOW - INVESTING_NET_CASHFLOW` where applicable.

Finished state must reject rows where required Cashflow fields are blank. Calculation/result messages must include the cash-equivalent mismatch condition when `END_BALANCE` differs from Balance `CASH_EQUIVALENT`. Derived fields are backend-calculated and MUST NOT be trusted from the client payload on save. [ev→R6]

### R7 Save draft/Finished as full replacement in one transaction - 強制點: both
covers-prd: FR-008

Save and Finished use the same save endpoint. The backend must validate `applicationNo` and `isFinish`, load loan summary, validate fields according to draft vs Finished mode, save the main record, delete existing Balance/Income/Cashflow rows for the `applicationNo`, insert the submitted rows in request order, and update the checkpoint in one transaction.

Finished requires at least one submitted annual GI row and allows up to five annual rows. Empty annual columns are not submitted and are not persisted; validation applies to every submitted Balance/Income/Cashflow row, but the contract MUST NOT require all five years to be filled. Because save is delete-and-rebuild, reducing the submitted year count intentionally removes previously persisted years that are omitted from the payload.

Save and Finished requests MUST include a non-null `mainMap` and non-null `balanceMap`, `incomeMap`, and `cashflowMap` arrays. The three detail arrays MUST be non-empty, contain no null elements, have the same length, and contain one to five submitted annual rows. These shape checks run before any main/detail delete, insert/update, or checkpoint write. Missing maps, null lists, empty lists, null row elements, length mismatch, or more than five rows MUST return controlled validation/business errors and leave existing DB rows/checkpoint unchanged. Implementation and automated negative tests are code-stage DoD, pending RD.

Checkpoint polarity is `isFinish=false -> EPROC00116 = "Y"` and `isFinish=true -> EPROC00116 = "N"` in the `TB_CHECK_POINTS_CS` or `TB_CHECK_POINTS_CU` table. Any failure after delete must roll back the main/detail/checkpoint writes. Save is not safe for automatic client retry.

Calculation is a read-only business operation: `epl-calc-c0-financial-statement-comments` MUST return only derived Balance/Income/Cashflow maps and result messages, and MUST NOT update `TB_CHECK_POINTS_CS`, `TB_CHECK_POINTS_CU`, main, Balance, Income, or Cashflow tables.

Finished may only proceed after successful Calculation and a balanced result: Balance `DIFFERENCE` must be zero and Cashflow `END_BALANCE` must equal Balance `CASH_EQUIVALENT`. To-be BE save MUST harden `isFinish=true` by recomputing or verifying the submitted/calculated Balance `DIFFERENCE` and Cashflow `END_BALANCE` against Balance `CASH_EQUIVALENT` before any completed checkpoint or main/detail mutation. Direct imbalanced Finished payloads MUST return controlled validation/business errors and leave existing main, Balance, Income, Cashflow, and checkpoint rows unchanged. Implementation and automated negative tests are code-stage DoD, pending RD. [ev→R7]

### R8 Export Excel and PDF only from complete persisted data - 強制點: both
covers-prd: FR-009, FR-010

PDF and Excel export endpoints must require `applicationNo`, load loan summary, main record, and all three GI detail groups from persisted data, and fail with `FAILED_E116` or platform-equivalent download error when any required group is missing. Export must be POST and must not rely on unsaved UI state. Successful Excel uses a binary download response; successful PDF uses the platform PDF/download response.

Export visibility MUST align with R1 `haveData`: Print/Excel/PDF actions may be shown from persisted current-case state only when main, Balance, Income, and Cashflow are all present. Missing Cashflow is incomplete persisted data, not a printable state.

c0 public API and output ownership remain `EPROC00116` through `epl-ppdf-c0-financial-statement-comments` and `epl-pxls-c0-financial-statement-comments`. Existing i0/`EPROI00116` template paths and names are approved as shared report assets for this bundle; this SRS does not require a c0-specific template rename. Exported content must come from c0 persisted GI data and no i0 public API route may be exposed. Future report-service R2/template redesign may rename assets without changing this API contract. [ev→R8]

### R9 Preserve GI downstream contract and checkpoint routing - 強制點: both

### R10 Enforce validation, authorization, safe errors, and safe logging - 強制點: both
covers-prd: FR-001, FR-002, FR-007, FR-008, FR-009, FR-010

All seven endpoints require platform authorization using their exact `TB_API_AUTH.API_ID` rows. `epl-pxls-c0-financial-statement-comments` MUST have a `TB_API_AUTH` row before code-stage approval, using `API_ID=epl-pxls-c0-financial-statement-comments` and `REF_FUNCTION_ID=EPROC00116`; its `ROLE` is copied from the existing c0 `epl-ppdf-c0-financial-statement-comments` row (same export action → pxls roles == ppdf-c0). RD/DBA code-stage DoD requires applying the seed and rerunning DB/API authorization checks before `rd-done`.

`TB_API_AUTH` endpoint authorization is necessary but not sufficient. Backend service logic MUST also enforce case/page authorization for the requested `applicationNo` and EPROC00116 page before business execution. Info, reference query, PDF, and Excel endpoints require backend case/page read access; calculation, draft save, and Finished require backend case/page edit access. Save/Finished authorization MUST run before field validation, main/detail delete or insert/update, and checkpoint writes. Export authorization MUST run before loading/generating/download-returning report files. Unauthorized direct API calls MUST return platform authorization/access-denied errors and leave main, Balance, Income, Cashflow, checkpoint, and generated report artifacts unchanged. FE pageAuth/button filtering remains UI control only; implementing the backend guard and automated negative tests is code-stage DoD, pending RD/Security.

Request validation must produce controlled EPRO/API errors for blank `applicationNo`, blank `isFinish`, missing calculation lists, invalid numeric/date fields, missing Finished fields, no-data, save failure, calculation failure, and download/export failure. `E130` is the malformed request-validation code for invalid save/calc-style payloads and must be returned only from those validation branches, not from successful business processing.

The contract must recognize current service errors and compatibility aliases: `MSG_DATA_NOT_FOUND`, `MSG_OVER_COUNT_LIMIT`, `MSG_QUERY_FAIL`, `MSG_ERROR_APPLICATION_NO`, `E130`, `COMMON_MSG_TOTAL_FAIL`, `COMMON_MSG_ERROR_LON`, `COMMON_MSG_PRINT_FAIL`, `COMMON_MSG_DOWNLOAD_SUCCESS`, `COMMON_MSG_DOWNLOAD_FAIL`, `COMMON_MSG_SAVE_SUCCESS`, `COMMON_MSG_SAVE_FAIL`, `MSG_EXPORT_SUCCESS`, `COMMOM_MSG_NO_DATA`, `EPROI0_0160_MSG_ERRO_CURRENCY`, `EPROI00119_MSG_VALIDATE_004`, `E116` / `FAILED_E116`, and `E126` / `FAILED_E126`. Business errors use the EPRO `code/message/data` envelope with HTTP 200; HTTP 401 is reserved for authorization errors such as `E405`/`E498`. Logs must not include full financial statement payloads or sensitive personal data.

The public routing contract MUST expose only the `epl-*` endpoints in this SRS and MUST NOT require adapter routes or action inputs named `inti`, `search`, `calculation`, `save`, `print`, or `printPDF`. Legacy typo/message keys such as `COMMOM_MSG_NO_DATA` and `EPROI00119_MSG_VALIDATE_004` remain allowed only as compatibility aliases or mapping provenance; the canonical migrated reference-query no-data error remains `E116` / `FAILED_E116`. [ev→R10]

### R11 Recalculate after amount edits and hide export actions - 強制點: FE
covers-prd: FR-007, FR-009, FR-010

After a successful Calculation, any user edit to Balance, Income, or Cashflow amount fields must mark the page as needing Calculation again, show/enable the Calculation action for editable users, hide Print/Excel and PDF actions, and block Finished until Calculation is rerun successfully. Reference-application copy also counts as data replacement and must follow the same "needs Calculation / hide Print/PDF" behavior before Save/Finished.

Print/Excel and PDF may only be visible again when the page has complete persisted data per R8 and the current unsaved UI state is not marked as needing recalculation. This carries PRD BR-006 and AC-005.

The same FE state must preserve BR-016 currency formatting: changing currency calls `setNumberIsDecimal`, KHR amount fields show/input zero decimals, and USD fields show/input decimal formatting. [ev→R11]
