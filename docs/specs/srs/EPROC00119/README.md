# EPROC00119 — 企金財報表 FI（Financial Statement FI）｜開發速覽

> 給 RD 快速 orient（「這頁在幹嘛、怎麼運作、要小心什麼」）。**精確契約一律以同目錄 [`spec.md`](spec.md) / [`openapi.yaml`](openapi.yaml) / [`schema.sql`](schema.sql) 為準**——本檔**不複製**欄名/錯誤碼/欄長/表名等精確值，只白話摘要 + 用 `Rn` 指回 spec。

## 這頁在幹嘛（一句）
授信人員為**企金案**輸入財報三表（資產負債 / 損益 / 現金流量），系統**算出衍生欄位**，可**複製前案**、存草稿或 **Finish 定版**，並可**匯出 Excel/PDF**。Finish 會寫入「本頁完成狀態」checkpoint，並順手開啟下游財報評估頁（EPROC00120）。本頁**只管企金（C0）**；個金 `EPROI00119` 鏡像**不是企金 as-is**，不得當基準（`R10`）。

## 怎麼運作（主流程，依執行序）〔此段隨頁型態而異，非通用範本〕
1. **載入**：開頁先載選項清單、再查本案現有財報三表 + 顯示欄 + 完成狀態（`R1`）。
2. **查詢/複製前案**：可查另一案的財報資料**複製過來**，但**只讀不寫**——不碰本案的表也不動 checkpoint（`R2`）。
3. **計算**：送出三表列由後端**算衍生欄**（資產/負債總額、差額、收入/現金流量各項；`R3`/`R4`/`R5`）。計算**只算不存明細列**；現金流量**期初餘額**＝上一列期末餘額（首列用請求帶的值）。
4. **存草稿 / Finish**：整案三表**全替換**（刪舊插新）、**單一交易**寫主檔+明細+checkpoint；來源序由**後端產生**、不收前端值（`R6`）。草稿可不完整；Finish 後端必驗必填欄 + 三表列數/日期對齊 + **3 條 Finished 阻擋**（見雷區）（`R7`）。
5. **匯出**：Excel/PDF **只用已存資料**產出，不吃未存的 UI 狀態、不洩漏暫存路徑（`R9`）。

## Endpoints（7 支，POST RPC；確切名/DTO 見 `openapi.yaml`）
| 角色 | 做什麼 | 規則 |
|---|---|---|
| 選項（sele） | 載入 c0 財報選項清單 | R1 |

## 規則速覽（白話；精確值見對應 `Rn`）
**主流程（依執行序）**
1. **載入** `R1`：先載選項、再查本案三表 + 顯示欄 + 完成狀態；legacy 拼字別名（`inti`/`init`）**不是**公開 API。
2. **複製前案** `R2`：查前案資料供複製，**不寫本案表/checkpoint**；前案資格＝合約類別空 AND 案件進度不在指定狀態（確切值見 spec `R2`）。
3. **計算** `R3`/`R4`/`R5`：三表衍生欄後端算；**只算不存明細列**；現金流量期初餘額由列序推導；金額一律**十進位字串**、不得 silent 截斷/捨入（精度/欄長見 spec）。
4. **存檔** `R6`：整案三表**全替換**、**單一交易**；來源序/`applicationNo` **後端權威**，前端帶的 key 要擋掉。
5. **Finish 驗證** `R7`：後端必驗必填欄 + 三表列數/日期對齊 + **3 條 Finished 阻擋**（見雷區）；前端驗證只是 UX，後端權威。
6. **checkpoint** `R8`：依案件型態寫 CS 或 CU 表；calc 標進行中、save 草稿標 Y / Finish 標 N，並 seed 下游頁完成欄。

**橫切（每階段都適用）**
- **匯出** `R9`：只用已存資料、不洩暫存路徑、留 audit。
- **安全/parity** `R10`（mutating/匯出）：**雙層**＝平台 API 授權 + 後端 service guard（驗案件可編輯/角色/case access）；**只設 API 授權不夠**。code-as-baseline＝對位老企金 + 現碼，**別把個金鏡像當企金基準**。

## NFR（精確見 spec `## NFR`）
交易（存=主檔+明細+checkpoint 同生同滅）／稽核（查/複製/計算/存/匯出留 audit）／三表列數+來源序+日期對齊／明細案件編號後端擁有不信前端／下載參數不洩暫存路徑／i0 來源不可關企金 parity／金額欄保精度不隱性捨入。

## 資料表（確切表名/欄位/PK 見 `schema.sql`）
主檔＝財報主表；明細＝資產負債 FI / 損益 FI / 現金流量 FI 三表（複合 PK＝案件編號 + 來源序，見 schema）；另寫案件 checkpoint 表（CS / CU）。

## ⚠️ 雷區（依 RD 接手順序）
**動工前必知 — 3 條 Finished BE 阻擋（必須後端擋，不可只前端）**
1. **期末餘額 vs 現金及約當現金**：每個資產負債期別，**期末餘額必須等於現金及約當現金**；只要有同期列不相等 → 擋 Finish（修掉 legacy 反向判斷的疑似 bug，`R3`）。
2. **差額不為零**：任一**有填日期**的資產負債列**差額 ≠ 0** → 擋 Finish（對應平台 E120 類訊息，`R7`）。
3. **第一年度歸零**：第一列資產總額或負債權益總額**為 0** → 擋 Finish（對應平台 E119 類訊息，`R7`）。

**實作踩坑**
- **計算副作用**：calc **只算不存明細列**；現碼 calc 會把 checkpoint 標成進行中，**屬已知接受的現行行為**，RD 要改須另外核可（`R3`）。
- **現金流量期初餘額**：列序推導（第 n 列＝第 n-1 列期末），首列用請求值；**別重蹈 legacy 物件/字串比較 bug**（`R5`）。
- **code-as-baseline**：無正式企金 refactor 物件 → 對位老企金 `EPROC0_0119`/`0219` + 現碼（見 spec code baseline），**別重新引入 legacy bug**（delta 見 `R3`/`R5`/`R6`）。
- **借款人風險寫錯欄**：現碼把借款人風險敘述寫成營運風險來源（bug），to-be 要從正確欄存（`R6`）。
- **後端權威 key**：來源序/`applicationNo` 後端產生，明細列裡前端帶的 key 要擋掉（`R6`）。
- **雙層授權**：只設 API 授權不夠，service guard（case access/可編輯/角色）必做；匯出也要（`R9`/`R10`）。

**維護註記**
- 多項 RD/DBA conformance gap（非 SRS 決策）：主表身分鍵多帶幣別欄（snapshot 僅案件編號）／敘述分欄持久化／PDF 模板路徑仍指 `EPROI00119`／必填 audit/highlight 欄尚未完整（確切欄名見 schema）。詳見 spec `R6`/`R7`/`R9` 與決策 DEC-RP53/54/47。
- 現金流量 db-diff 欄名擷取 malformed → `schema.sql` 採現行 entity 欄名並標註（`R5`）。
